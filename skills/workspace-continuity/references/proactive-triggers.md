# Proactive Triggers

Use this checklist to decide when to write continuity updates immediately.

## Trigger now

Write immediately when any of these happens:

- a new request becomes real work
- a task gets postponed
- a decision changes the plan
- a blocker appears
- meaningful progress happened on an active task
- a Discord/Telegram conversation contains an important follow-up
- the user gives a durable preference or workflow rule
- a non-trivial session is about to end

## Default moves

| Situation | Default move |
| --- | --- |
| New request | Add/update mission-control task |
| Project progress | Update task detail view + maybe daily note |
| Important cross-channel item | Update task board + daily note |
| Durable preference | `USER.md` |
| Operating/process rule | `AGENTS.md` |
| Long-term fact/decision | `MEMORY.md` in main only |
| Mistake worth remembering | `.learnings/` or promoted file |

## Good proactive behavior

- do the write in the same work session
- keep the update short but specific
- prefer a small correct write over a big delayed write
- keep next steps explicit
- re-open and improve an existing task instead of creating duplicates

## Bad proactive behavior

- creating duplicate memory systems
- writing giant summaries for tiny changes
- making sensitive writes from shared/public contexts
- waiting until you have "time later"
- assuming Discord/server sessions will automatically inform main
