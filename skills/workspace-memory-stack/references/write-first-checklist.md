# Write-First Checklist

Write before responding when most of these are true:

- the detail is easy to lose
- it changes the current plan
- it would be annoying to reconstruct later
- it belongs to active work, not just general chatter
- the write is internal and low-risk

## Good write-first cases

- new blocker
- new deadline
- important decision
- task got postponed
- cross-channel request that must be mirrored into main memory
- active task gained meaningful progress or new next steps

## Do not overdo it

Do not stop to write for:
- tiny acknowledgements
- casual banter
- obvious transient chatter
- duplicate information already captured clearly elsewhere

The point is durability, not ritual.
